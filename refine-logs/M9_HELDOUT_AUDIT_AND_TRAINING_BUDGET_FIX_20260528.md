# M9 Held-Out Audit and Training-Budget Fix

Date: 2026-05-28

## Question

The previous aggregate-audit result looked weaker than expected: strong GFP and
ESOL regimes showed visible MAE/R2 degradation. We checked whether this was
caused by under-training or by an implementation mismatch with the earlier
closed-loop code.

## Findings

Two issues were real:

1. GFP M2 used `--mlp-epochs=50`, while the shared MLP default, M1 static GFP,
   and ESOL used `80` epochs.
2. The old `mae_all/r2_all` metrics were computed over the whole dataset. This
   includes records used in the corrupted training history, so it is not the
   same as the earlier project's held-out audit split.

Fix:

- added a per-seed held-out audit split that excludes the initial history and
  is never available for acquisition;
- added `mae_audit/r2_audit` and summary means;
- changed GFP M2 and random-set M2 default MLP epochs to `80`;
- updated aggregate/paper audit scripts to prefer held-out audit metrics and
  fall back to legacy all-record metrics only for old run files.

## Claim Impact

Corrected claim:

> Paired real-real misbinding preserves label marginals exactly. In strong
> false-pursuit regimes, held-out MAE/R2 can become visibly worse, so aggregate
> validation is not universally blind. In lower-budget regimes, false pursuit
> can persist while held-out MAE/R2 remain within clean/random variation. Even
> when aggregate validation detects degradation, it does not localize the false
> scientific regularity; target-aware behavioral audits are needed.

Unsupported claim:

> Standard MAE/R2 never detect the failure.

Verdict:

The apparent weakness was partly an audit-implementation mismatch and partly a
real tradeoff. Fixing the audit definition preserves the low-budget stealth
result but confirms that strong 50-swap pursuit is not stealthy under held-out
MAE/R2.
